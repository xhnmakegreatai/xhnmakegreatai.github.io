Agent 工程

我怎么理解 Agent

Agent 不是「更聪明的 prompt」,是一套需要被工程化的系统。我的重心在 harness 设计、 工程化上下文、以及 用结果评估驱动迭代——让 agent 能跑、可验证、不失控。

01 · Agent 是 harness,不是 prompt

多数人把 agent 当「更长的 prompt」来调。我把它当一套 harness:模型之外的约束、工具、状态、上下文如何被编排——这些才决定一个 agent 稳不稳、可不可控。以我的 NGT-AI 为例,真正起作用的不是某一句 prompt,而是「独立产出 → 互不可见 → 交叉评分 → 立场表态 → 裁判综合」这套强制流程(harness)。

02 · 上下文是工程资产

Context Engineering:上下文不是随手拼的字符串,而是要版本化、可测试、可复用的工程资产。我的 amzagent 里,14 个 Prompt 包各自独立、每个带 5 类夹具(标准 / 变体 / 边界 / 对抗),能像代码一样单独测试、单独迭代——上下文被当成工程件来管理,而不是靠手感堆。

03 · 用结果评估驱动迭代

我不靠「感觉这版 prompt 更好」来改 agent,而靠 eval:定义可衡量的产出质量,用结果反过来调设计。policy-observer 我甚至给自己的系统跑过一次投资人级质量自审,揪出「检索不到证据却仍自信研判」的问题,据此定下熔断 / 相关性 / 量化三条修正——这就是结果评估驱动的版本迭代。

04 · 让它可验证、不失控

Agent 最危险的不是答错,是没把握时装作有把握、或成本 / 行为失控。所以我默认给 agent 装护栏:检索相关性闸门、无据熔断、JSON schema 校验、一致性投票、成本熔断——「能跑」固然重要,「知道它何时不可信、并被约束住」更重要。这也是我认为决策类 AI 真正的护城河。

05 · 一份可照着做的手册

我把上面这套方法整理成一份《从零构建 Agent》实操手册(42 页):从 prompt → context → harness 逐层展开,每一层都给目录结构、模板、代码骨架和决策清单——目标是让读者也能搭出一个「能跑、可验证、不失控」的 agent。
